## 问题 1（P0）：soft-matte 暗色饱和键色崩溃

### 现象

`remove_chroma_key.py --key-color 7f0000 --soft-matte` 直接抛出 traceback 而非契约约定的中文错误：

```text
File "...remove_chroma_key.py", line 268, in _apply_alpha_to_image
    spill_strength = source_rgb[spill[0]]
IndexError: list index out of range
```

（已实际复现，exit code 1 + 完整 traceback。）

### 根因

校验条件与算法前置条件不一致：

- `_validate_args`（74–79 行）只要求 `max(key) - min(key) >= 96`；
- `_spill_channels`（124 行）要求最强通道 `>= 128`，否则返回空列表 `[]`；
- 于是 max 通道落在 **[96, 127]** 的暗色饱和键（`#7f0000` 暗红、`#600030` 暗洋红等）通过校验，却在 soft-matte 分支无保护地索引 `spill[0]`；
- `--auto-key` 路径有同样缺口：`_sample_border_key`（462 行）使用相同的 96 阈值；
- 现有测试 `test_low_saturation_key_rejects_soft_matte_and_despill` 只覆盖了 `#808080` 灰键，未覆盖该区间。

### 修复方案（三处防线）

1. **`_validate_args`（74–79 行）**：把饱和度校验从单一条件改为复合条件：

   ```python
   key = _parse_key_color(getattr(args, "key_color", "#00ff00"))
   if max(key) - min(key) < 96 or not _spill_channels(key):
       _die("--soft-matte/--despill 只支持具有主导亮通道的高饱和色键；灰、白、黑或过暗的键色不受支持。")
   ```

   等价性验证：`#808080` 仍因 `max-min < 96` 被拒（现有测试不受影响）；`#7f0000` 因 spill 为空被拒；`#008000`（max=128 边界）spill=[1]，正常放行。

2. **`_sample_border_key`（462 行）**：同样把 `max(candidate) - min(candidate) < 96` 扩展为复合条件，保证 auto-key 路径报出受控错误。

3. **`_apply_alpha_to_image` soft-matte 分支（266 行后）**：加防御性保护，使崩溃在结构上不可达（即使未来校验入口变更）：

   ```python
   spill = _spill_channels(key)
   if not spill:
       _die("色键缺少主导亮通道（最强通道 < 128），--soft-matte/--despill 不受支持。")
   ```

   注意：保护只加在 soft-matte 分支；hard-key + `--despill` 对空 spill 的合法语义是无操作，不改动。

### 测试计划（`tests/test_remove_chroma_key.py`）

- `#7f0000` + `--soft-matte` → `SystemExit`，stderr 为受控中文错误而非 `IndexError`；
- `#7f0000` + `--despill` → `SystemExit`；
- 暗红色边框图 + `--auto-key border --soft-matte` → 受控 `SystemExit`，且不写出文件；
- `#008000` 边界键 + `--soft-matte` → 正常处理（回归保护）；
- 保留 `#808080` 拒绝用例不变。

## 问题 2（P0）：`_validate_generated_image` 死变量与提前求值

### 现象与根因

`image_gen.py:1236`：

```python
profile = getattr(args, "model_profile", resolve_model(getattr(args, "model", None))[0])
```

- `profile` 赋值后在函数内（1237–1254 行）**从未被使用**；
- `getattr` 的默认值会被**提前求值**：即使 `args.model_profile` 存在，每次校验生成图都会无条件调用一次 `resolve_model()`。使用 `--allow-model-fallback` + 未知模型名时，会在生成完成后向 stderr 重复打印 fallback warning；直接以非法 model 调用该函数时还会意外 `die()`。

### 修复方案

直接删除该行（一行改动）。函数内其余逻辑只依赖 `args.size`，不涉及 `profile`。

### 测试计划

无需新增测试；现有 `test_generated_*` 系列（格式/尺寸/档位校验）回归通过即可。

## 问题 3（P1）：`_atomic_write` 硬链接兼容性

### 现象与根因

`image_gen.py:1272` 与 `remove_chroma_key.py:519` 用 `os.link(temporary, path)` 实现 no-clobber 原子写入。在不支持硬链接的文件系统（exFAT U 盘、部分网络盘/云同步目录）上抛 `OSError`：

- 对 `image_gen.py`，失败发生在**计费 POST 之后** → 请求被标记 `ambiguous` 并加状态锁 → 此类环境下**每次真实请求都必然落入 ambiguous**，生成成功也无法正常交付；
- 对 `remove_chroma_key.py`，直接报错无法写出。

### 修复方案

`os.link` 失败（非 `FileExistsError`）时退化为排他创建 `open(path, "xb")`：

```python
else:
    try:
        os.link(temporary, path)
    except FileExistsError:
        die(f"保存前检测到输出文件已被创建，拒绝覆盖：{path}")  # 保持原有 no-clobber 语义
    except OSError:
        # 不支持硬链接的文件系统（exFAT、部分网络盘）退化为排他创建；
        # no-clobber 保证不变，崩溃原子性在该类文件系统上无法更强。
        try:
            with path.open("xb") as target:
                target.write(content)
                target.flush()
                os.fsync(target.fileno())
        except FileExistsError:
            die(f"保存前检测到输出文件已被创建，拒绝覆盖：{path}")
        except OSError as exc:
            die(f"无法保存生成图片：{path}（{exc}）")
        temporary.unlink(missing_ok=True)
        temporary = None
```

两个脚本同步修改（`remove_chroma_key.py` 用 `data` 参数，结构相同）。语义权衡：xb 直写在崩溃中途可能留下不完整文件，但仅在本来就不支持硬链接的环境启用，no-clobber 保证不受影响。

### 测试计划

- 两个脚本各加：monkeypatch `os.link` 抛 `OSError` → 输出仍成功写入且内容完整；
- 目标已存在 + link 失败 → 仍报"拒绝覆盖"且不改动已有文件；
- `image_gen.py` 侧确认 fallback 路径发生在状态机内时状态分类不变（保存失败仍标 `ambiguous`，但 fallback 成功后走正常 `else: state_path.unlink()` 成功路径）。

## 问题 4（P1）：色键脚本死代码清理

### 现象

以下函数经 grep 验证**无任何调用方**（只在彼此之间互相引用），是被向量化 ImageChops 实现替换前的旧逐像素标量实现：

|---|---|---|
| `_smoothstep` | 109–111 | 仅被 `_soft_alpha` 引用 |
| `_soft_alpha` | 114–119 | 引用已废弃的距离阈值语义 |
| `_single_spill_colored_alpha` | 139–152 | 仅被 `_dominance_alpha`/`_is_key_edge` 引用 |
| `_dominance_alpha` | 155–163 | 无调用 |

| `_despill` | 174–183 | 无调用（与 `--despill` 标志同名，易误导） |
| `_recover_foreground` | 186–195 | 无调用 |

保留：`_spill_channels`、`_key_dominance`、`_clamp_channel`、`_channel_distance`（均为现网代码）。

### 附带问题

### 修复方案

1. 删除上表 7 个函数（约 60 行）；
2. 删除阈值顺序校验，**保留**范围校验（`test_nan_threshold_is_rejected` 依赖 NaN 拒绝，须继续通过）；
3. `chroma-key.md` L32 补一句"仅保留范围校验"；
4. 实施前 grep 确认无测试依赖顺序校验，若有则同步删除该断言。

### 测试计划

全量回归即可；行为零变化（死代码 + 无意义校验移除）。

## 问题 5（P2）：prompt 临时文件 ID 文档与正则不一致

### 测试计划

现有 prompt 文件契约测试回归通过即可。
